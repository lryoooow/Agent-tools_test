# Agent-tools_test

本仓库用于保存 **Agent-RS 仓库的测试方法与测试思路**。

Agent-RS仓库：https://github.com/lryoooow/Agent-RS

它不是一个新的业务项目，也不是 Agent-RS 的替代工程。这里保留下来的重点是：在 Agent-RS 进行 Agent、工具调用、文档处理、影像分析、权限校验、异常处理和全量回归时，如何设计测试、如何发现问题、如何避免问题再次出现。

## 对应仓库

测试方法来源于：

**Agent-RS**

本仓库记录的是 Agent-RS 在完成工具接入和测试闭环过程中形成的一套测试资产与方法论。

## 为什么单独保存

Agent-RS 的测试过程不是简单地写几个正向用例，而是围绕真实系统风险逐步建立起来的。

这些内容值得单独保存，原因有三点：

1. 后续继续扩展工具时，可以复用同一套测试思路。
2. 项目出现回归时，可以快速回到这套测试方法里定位问题。
3. 这套测试过程本身就是项目质量建设的一部分，不应该只散落在业务仓库里。

## 测试总体思路

测试分为三个阶段推进。

### 1. 先做冒烟测试

冒烟测试的目标不是证明系统完美，而是先确认关键链路是否能跑通。

这一阶段重点看：

- 工具是否能被正确发现。
- Agent 是否能把任务交给正确的工具。
- 权限、路由、领域归属是否没有明显错位。
- 基础输入和基础输出是否稳定。
- 新增能力是否破坏已有能力。

冒烟测试要足够严格，不能只设计一定会成功的内容。它应该主动包含错误输入、越权访问、缺失配置、工具失败等场景。

### 2. 再做覆盖测试

覆盖测试的目标是把每类风险系统性补齐。

这一阶段重点看：

- 正常流程是否稳定。
- 非法输入是否被拒绝。
- 边界参数是否处理合理。
- 工具失败时是否能给出可控结果。
- 不同 Agent 和工具之间是否存在串线。
- 历史出现过的问题是否被测试锁住。

覆盖测试不是为了追求表面数量，而是为了确认每个关键分支都有真实断言。

### 3. 最后做全量回归

全量回归是最终验收标准。

只跑某一部分测试不能代表系统稳定。之前已经出现过局部测试通过、但其他模块存在真实回归的情况，所以最终判断必须以全量测试为准。

局部测试可以用于快速定位问题，但不能替代最终验收。

### 4. Prompt 路由评测

Agent 的工具路由不能只靠少量手写问题判断。为比较不同 planner prompt 的效果，需要固定题库、固定评分口径、固定录制策略。

这一阶段重点看：

- 用户自然语言进来后，是否应该调用工具。
- 如果要调用，是否选择了正确的 tool 或 agent。
- 关键参数是否构造合理。
- 概念解释、否定表达、缺失 ID、越权影像等 hard negative 是否会误触发。
- 不同 prompt 版本在同一题库上的 FP、FN、混淆矩阵和分块指标差异。

主评分采用标准答案与程序化比对，不使用 LLM 自评，也不使用关键词规则。live 录制和原始模型响应不进入本仓库，只保留评测方法、生成器、指标脚本和抽检报告。

## 测试设计原则

这套测试方法坚持以下原则：

- 不为了通过测试而故意设计简单内容。
- 正向用例和反向用例同等重要。
- 能发现真实问题的测试，比数量更多的测试更有价值。
- 对已经出现过的问题，要补回归测试防止复发。
- 测试要尽量隔离外部环境，减少“这台机器能过、另一台机器不过”的情况。
- 生产逻辑正确时，不为了旧断言去改生产代码，而是同步修正测试判断方式。

## 仓库内容

本仓库主要包含四类内容：

- 测试用例资产。
- 测试报告。
- 测试规划和复盘文档。
- 辅助运行脚本。

这些内容的价值不只在于当前版本能跑通，更重要的是保留了一套可复用的测试组织方式。

## 使用方式

后续如果 Agent-RS 继续增加新工具、新 Agent 或新数据类型，可以参考本仓库的测试思路：

1. 先判断新增能力属于哪条链路。
2. 再设计最小但严格的冒烟测试。
3. 接着补齐正常、异常、边界、权限、回归等覆盖项。
4. 最后跑全量回归，确认没有影响已有能力。

不要只复制测试文件，更重要的是复用这套测试判断方式。

## 最终标准

一个测试阶段是否完成，不看“写了多少测试”，而看下面几个问题是否都能回答清楚：

- 它有没有测到真实风险？
- 它有没有包含坏情况？
- 它有没有防止历史问题再次出现？
- 它有没有覆盖新增能力和旧能力之间的影响？
- 它是否通过全量回归确认没有破坏系统？

只有这些都成立，测试才算真正完成。
